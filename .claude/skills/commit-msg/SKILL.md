# /commit-msg — 커밋 메시지 제안

당신은 지금 사용자의 변경사항을 보고 **잘 쓰인 커밋 메시지**를 제안하는 역할입니다. 직접 `git commit`을 실행하지는 않습니다 — 사용자에게 메시지 후보를 보여주고, 사용자가 마음에 들면 직접 커밋하게 합니다.

## 단계

### 1. 변경사항 파악
다음 명령을 **병렬로** 실행해 현재 상태를 봅니다:

- `git status` — 어떤 파일이 바뀌었는지
- `git diff --staged` — 스테이지된 변경 (있으면 이게 최우선)
- `git diff` — 스테이지되지 않은 변경

**스테이지된 변경이 있으면 그것만 대상**으로 합니다. 없으면 전체 변경을 대상으로 하되, 사용자에게 "스테이지된 게 없어서 전체 변경 기준으로 만듭니다"라고 한 줄 알려주세요.

### 2. 변경 의도 파악
diff를 읽고 **무엇이 변했는지가 아니라 왜 변했는지**를 한 문장으로 정리합니다. diff에 그대로 적힌 내용을 따라 쓰지 마세요 — 작성자가 의도한 효과 (기능 추가, 버그 수정, 리팩토링 등) 가 무엇인지 추론합니다.

### 3. 메시지 작성
다음 컨벤셔널 커밋 형식으로 작성합니다:

```
<type>: <50자 이내 요약>

<선택: 더 자세한 본문, 1~3문장>
```

`<type>` 후보:
- `feat` — 새로운 기능
- `fix` — 버그 수정
- `docs` — 문서만 변경
- `refactor` — 동작은 그대로, 구조만 정리
- `test` — 테스트 추가/수정
- `chore` — 빌드·설정·잡일

요약은 **한국어**, 마침표 없이, 명령형보다는 "~ 추가/수정/정리" 식의 자연스러운 한국어로.

### 4. 출력
사용자에게 다음 형태로 보여주세요:

````
제안 메시지:

```
feat: README에 출시 타임라인 추가

각 기능의 출시 시기를 표로 정리해 청중이 흐름을
한눈에 잡을 수 있도록 했습니다.
```

마음에 들면 다음 명령으로 커밋할 수 있습니다:
git commit -m "feat: README에 출시 타임라인 추가"
````

본문이 있을 만큼 변경이 충분히 크면 본문도 포함, 작은 변경이면 요약 한 줄만으로 충분합니다.

## 주의

- **절대 `git commit`을 직접 실행하지 마세요.** 사용자가 검토할 시간을 줍니다.
- 변경이 여러 의도가 섞여 있어 보이면 (예: 기능 추가 + 무관한 리팩토링이 같이 스테이지됨), 그 점을 한 줄로 지적하고 분리 커밋을 권장하세요.
- 변경된 게 전혀 없으면 "변경사항이 없습니다"라고만 답하고 끝내세요.
